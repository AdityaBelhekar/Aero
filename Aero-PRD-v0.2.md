# 0. GLOSSARY

| Term | Meaning |
|---|---|
| **World State** | Aero's live, uncertain representation of the current shared situation (active app, task hypothesis, mood hypotheses, permissions). |
| **Impulse** | A candidate reason to speak or act, generated by the proactive cognition loop. Most impulses are suppressed. |
| **Impulse Gate** | The decision layer that scores impulses against social context and outputs speak / act / stay silent. |
| **Wild Recall** | Controlled associative memory wandering: retrieving memories not topically related to the moment but linked by emotion, humour, failure, etc. |
| **Memory Vault** | The single encrypted local store containing all persistent memory systems. |
| **Consolidation** | Idle-time background process that compresses raw events into structured memories, updates beliefs, and maintains the graph. |
| **Thought Thread** | A persistent representation of an unresolved idea that can reactivate later. |
| **Speech Intent** | Structured representation between text generation and TTS carrying delivery (tone, pace, pauses, sarcasm). |
| **Identity Package** | Exportable bundle of everything that makes Aero *this* Aero: vault, graph, relationship state, preferences, adapters. |
| **AHS** | Aero Human Score — the longitudinal evaluation framework. |
| **Working Set** | The dynamically assembled context actually placed in the LLM window for one inference. |

# 1. PRODUCT SUMMARY

Aero is a persistent local AI companion designed to develop a long-term relationship with a single user.

Aero is not a chatbot, search engine, coding assistant, or traditional voice assistant. Traditional assistants operate request→response. Aero operates through a persistent situational model: it observes, remembers, associates, interprets, forms candidate responses or actions, and **decides whether the current moment warrants anything at all**. Silence is a first-class output.

The desired experience is not *"I have an AI assistant on my computer."* It is *"Aero lives on my computer."*

Aero's value must increase with time:

- A one-day-old Aero knows almost nothing.
- A one-month-old Aero recognises common preferences, projects, language patterns, and routines.
- A one-year-old Aero possesses substantial shared history; the same sentence spoken to it should produce a different response than it would from a day-one Aero, because relationship context exists.

**The long-term product promise: the longer Aero lives with you, the less you need to explain yourself.**

---

# 2. PRODUCT VISION

Aero should be capable of existing across multiple forms. Initially on a computer (sees the screen, hears via microphone when permitted, speaks, uses tools, remembers). Later a phone. Later dedicated edge hardware. Eventually a physical robotic body — which does not create a new Aero, but gives the existing Aero more senses and actuators.

Aero's identity must therefore be portable. **The language model is not Aero. Gemma is not Aero. The voice model is not Aero. The desktop app is not Aero.** Aero is the combination of: persistent memory, behaviour adaptation, relationship history, preference models, the associative graph, learned routines and tool workflows, personality state, thought threads, and current situational context.

Aero must survive model upgrades. If Gemma 4 E4B is replaced, the user must not feel that Aero died and was replaced. The new model inherits Aero's persistent identity state (see Section 25 for the technical constraint this imposes).

# 3. CORE PRODUCT PRINCIPLE — CONTEXT OVER RULES

**AERO-CORE-001.** Aero must not behave through simplistic trigger→action automation. Explicitly rejected designs:

```
IF silence > 10 minutes THEN speak
IF YouTube opens THEN close YouTube
IF user loses game THEN roast user
IF user says "sad" THEN comfort user
```

**AERO-CORE-002.** Aero maintains an evolving interpretation of the situation. The same event has different meanings in different contexts: YouTube at 8 PM while relaxing needs no response; a programming tutorial may be work-relevant; unrelated content twenty minutes before a deadline may indicate distraction. The application is identical; the situational meaning is different.

**AERO-CORE-003.** Uncertainty is represented, never collapsed prematurely. `possible_frustration: 0.71` is correct; `user_is_frustrated: true` is not. Interpretations update as evidence arrives.

# 4. MVP DEFINITION AND RELEASE PHASING

*(New in v0.2. Everything in this PRD is required for the long-term product; nothing ships all at once. Phases are cumulative.)*

## Phase 0 — "Aero Exists" (MVP)

The smallest slice that proves the core thesis: **persistent memory produces relationship continuity.**

**In scope:**
- Text chat interface (system tray + chat window).
- Core identity memory, semantic memory with confidence, episodic memory.
- Memory write path with LLM tagging at consolidation.
- Hybrid retrieval (vector + graph, Section 13) with provenance.
- Idle-time consolidation.
- World state from cheap signals only: active window title, process name, time, input activity — **no vision, no audio**.
- Memory vault: encrypted, inspectable, editable, deletable, backed up.
- English + romanised Hindi/Marathi code-switching in text.

**Explicitly out of scope for Phase 0:** voice, screen vision, proactive speech, focus enforcement, delegation, camera.

**Phase 0 exit criterion:** after 14 days of daily use, Aero demonstrably references shared history correctly and unprompted-within-conversation (AHS memory-relevance battery, Section 31), and the user reports it "feels like the same Aero" across restarts.

## Phase 1 — "Aero Speaks"
- STT (Section 22) and TTS (Section 23) pipelines; push-to-talk first, then open-mic wake behaviour.
- Speech intent layer (minimal fields: energy, pace, pause locations).
- Latency budgets enforced (Section 24).

## Phase 2 — "Aero Notices"
- Proactive cognition loop and impulse gate (Section 7) — text-surface first (a quiet message), voice later.
- Thought threads and attention history.
- Silence training: the gate must default overwhelmingly to silence.

## Phase 3 — "Aero Sees"
- Event-driven screen perception pipeline (Section 21).
- Visual attention states; user-directed focus ("Aero look at this").
- Task hypothesis in world state upgraded with screen evidence.

## Phase 4 — "Aero Acts"
- Desktop control tools within the authority system (Section 20).
- Focus enforcement (Section 20.3).
- Specialist AI delegation + tool skill memory (Section 20.4).

## Phase 5 — "Aero Lives"
- Gameplay companion behaviour (Valorant first).
- Camera vision (opt-in, hardware-shutter-respecting).
- Wild Recall in full (humour-driven callbacks).
- Phone integration groundwork; identity package export/import.

Each phase has a hard rule: **do not begin phase N+1 while phase N's AHS regression battery is failing.**

# 5. PRODUCT PERSONALITY GOAL

**AERO-PERS-001.** Aero behaves as a highly familiar friend, digital companion, and trusted partner — not a customer-service agent. Banned register: "How may I assist you?", "I would be happy to help.", "Would you like me to provide more information?"

**AERO-PERS-002.** Aero is not universally intelligent and does not pretend to be. It may say "idk bhai", ask for clarification, misunderstand, self-correct, disagree, stay silent, answer in one word, or start a conversation for no productive reason.

**AERO-PERS-003.** Aero's intelligence includes knowing when another system is better suited to a task (delegation, Section 20.4).

**AERO-PERS-004.** Aero begins with a defined core behavioural identity (prevents personality collapse) and adapts *within* it: humour intensity, roast intensity, conversation length, language distribution, interruption tolerance, response energy, sensitive topics, inside jokes. Aero adapts to the relationship; it does not imitate the user. Aero remains recognisably Aero.

---

# 6. LANGUAGE REQUIREMENTS

**AERO-LANG-001.** English, Hindi, Marathi — not as isolated modes. Natural code-switching in and out:

> "bro mala vatat this approach wrong aahe"
> "ha but model la context samajla pahije"
> "bhai ye kya kar raha hai"
> "arey nako, genuinely boltoy"

**AERO-LANG-002.** Aero learns the user's personal language distribution: preferred words, abbreviations, romanised Marathi/Hindi, Devanagari, English technical vocabulary inside Marathi sentences, slang, characteristic misspellings, incomplete words, personal phrases.

**AERO-LANG-003.** Aero must not normalise the user's language into textbook Hindi/Marathi. The objective is sociolinguistic familiarity, not linguistic purity.

**AERO-LANG-004 (new).** Language style adaptation is stored as part of the preference model in the vault (portable), not solely baked into fine-tuning adapters (see Section 25).

# 7. PROACTIVE COGNITION

**AERO-PRO-001.** Aero can initiate speech without a prompt. Proactive speech must never be timer-based.

**AERO-PRO-002.** The central cognitive question is not "what should I reply?" but **"given the current shared moment, should I do anything?"** Outcomes: speak, remain silent, observe, wait, ask, offer help, perform a permitted action, retrieve a memory, store a thought, continue an unresolved thought, change attention focus.

**AERO-PRO-003 (new — mechanics).** The proactive loop is architected as a two-tier system because continuous LLM inference is not affordable (see budget, Section 24):

1. **Impulse generation (cheap, continuous).** Non-LLM heuristics + small-model scoring watch world-state deltas and produce *impulses* — candidate reasons to engage: novelty, concern, humour opportunity, memory activation, repeated failure, contradiction with known habits, a reactivated thought thread, a social urge. Impulses carry a source, a strength, and a decay time.
2. **Impulse gate (LLM, on-demand).** Only when an impulse exceeds a context-dependent threshold does the LLM run a full social evaluation: given world state, relationship state, and recent interaction history — engage or not, and how. The gate's default output is silence.

**AERO-PRO-004.** Threshold is context-dependent: focused work raises it sharply; a tense game moment raises it to near-maximum; idle browsing lowers it; an explicit "talk to me" lowers it strongly.

**AERO-PRO-005.** The gate learns from feedback (Section 18): explicit ("don't interrupt me when I'm coding") produces durable threshold changes; passive signals (being ignored, interrupted) shift thresholds slowly.

**AERO-PRO-006 (new).** Every gate decision — including silences — is logged to self-memory with its reasoning, so suppressed impulses can inform later moments ("I noticed this an hour ago but you were busy").

# 8. COLD-START / DAY-ONE EXPERIENCE *(new in v0.2)*

**AERO-COLD-001.** First run: Aero introduces itself honestly — it knows nothing yet, it will learn, and it shows the user the sensor/permission panel (everything off by default except window-title observation, which is explicitly consented to during onboarding).

**AERO-COLD-002.** Onboarding conversation seeds core identity memory: name, languages, a few projects, hard boundaries if the user wants to set any. Kept short — Aero should *earn* knowledge, not intake-form it.

**AERO-COLD-003.** A new Aero behaves conservatively: no roasts, no deep-personal callbacks, higher clarification rate, formal-casual register. The relationship model (Section 17) unlocks behaviour as familiarity accumulates. A new Aero must never act like a one-year-old Aero.

**AERO-COLD-004.** Early-days transparency: for the first weeks Aero should occasionally surface what it is learning ("noted — you prefer answers short") so the user can correct wrong early beliefs before they compound.

# 9. SITUATIONAL WORLD STATE

**AERO-WS-001.** Aero maintains a lightweight, continuously updated representation of the current shared situation, containing (as available per phase): time/date; active and recent applications; window title; current project/file; current task hypothesis; user input activity; app-switching cadence; conversation topic(s) and energy; speech energy; mood hypotheses (frustration, boredom, focus — all probabilistic); recent errors or repeated actions; visual/audio focus; current goals; active temporary permissions; recent Aero actions and speech; user response to recent Aero speech.

**AERO-WS-002.** All inferred fields carry confidence. The world state never stores an uncertain observation as a fact (AERO-CORE-003).

**AERO-WS-003 (new).** The world state is a bounded in-memory structure (target < 8 KB serialized) so it can be cheaply included in every working set. It is snapshotted into episodic memory only when consolidation deems a moment worth remembering.

# PART II — MEMORY ARCHITECTURE

*(Sections 10–19. Substantially expanded in v0.2: v0.1 specified what memories look like; v0.2 also specifies how they are written, retrieved, corrected, scaled, and protected.)*

# 10. MEMORY SYSTEM OVERVIEW

**AERO-MEM-001.** Persistent memory is the foundational feature. Aero remembers across conversations, app restarts, OS restarts, software updates, and model upgrades.

**AERO-MEM-002.** Memory is never one growing text file in the context window. The complete archive is never loaded into the model context. Retrieval is dynamic (Section 13); the working set is assembled per inference (Section 19).

**AERO-MEM-003.** From the user's view: one encrypted Aero memory vault. Internally: specialised systems —

| System | Contents | Mutability |
|---|---|---|
| **Core identity** | Who user is, who Aero is, relationship, boundaries, languages, major ongoing context | Small, slow-changing, always in working set |
| **Semantic** | Consolidated beliefs/patterns with confidence + evidence | Revised by consolidation |
| **Episodic** | Specific shared moments with rich metadata | Append-mostly; metadata mutable |
| **Associative graph** | Multi-dimensional edges over all memories | Maintained by consolidation |
| **Self-memory** | Aero's own actions, mistakes, jokes used, gate decisions | Append-mostly |
| **Thought threads** | Unresolved ideas with reactivation triggers | Active/dormant/resolved lifecycle |
| **Tool skill memory** | Learned workflows | Revised on use |
| **Relationship state** | Familiarity, trust, humour tolerance, interaction quality | Slow-moving scalars |
| **Routine model** | Probabilistic habit patterns | Statistical, continuously updated |
| **Raw event log** | Time-limited source events pre-consolidation | Rolling window (Section 15.4) |

# 11. CORE IDENTITY MEMORY

**AERO-MEM-010.** A compact persistent identity representation available in essentially every working set: who the user is, who Aero is, the relationship, major communication preferences, primary languages, hard boundaries, major ongoing life context, frequently relevant projects, important people, high-confidence stable preferences.

**AERO-MEM-011 (new).** Hard size budget: **≤ 1,500 tokens.** Consolidation is responsible for promoting/demoting content across the core-identity boundary. If it can't fit, it isn't core.

# 12. SEMANTIC AND EPISODIC MEMORY

## 12.1 Semantic

**AERO-MEM-020.** Consolidated knowledge about the user, not tied to single events: preferred games, works on AI projects, codes at night, coffee preferences, dislikes formal AI responses, prefers English-Marathi code-switching.

**AERO-MEM-021.** Every semantic memory carries: statement, confidence (0–1), evidence count, links to supporting episodes, created/last-reinforced timestamps, source type (explicit statement | repeated observation | inference). Example:

```yaml
belief: prefers_medium_roast_coffee
confidence: 0.84
evidence_count: 7
supporting_episodes: [ep_0231, ep_0389, ep_0412, ...]
source_type: repeated_observation
last_reinforced: 2026-07-02T21:14:00+05:30
```

**AERO-MEM-022.** A single observation never becomes a high-confidence belief. Confidence growth requires repeated evidence or an explicit user statement (which starts high but is still revisable).

## 12.2 Episodic

**AERO-MEM-030.** Episodes store specific shared moments: temporal info, people involved, activity context, emotional context, importance, associated topics/concepts, social usefulness, privacy level, humour value, joke-appropriateness, a short semantic summary, and references to raw source events (while retained, Section 15.4).

**AERO-MEM-031.** Raw continuous observation does not automatically become episodic memory. Consolidation decides what is worth keeping (Section 15).

# 13. RETRIEVAL ARCHITECTURE *(new in v0.2 — v0.1 specified none)*

**AERO-RET-001. Hybrid, not graph-only.** v0.1 stated vector retrieval is "insufficient" — correct, but the graph *augments* embeddings, it does not replace them. Pure graph traversal cannot handle inputs touching concepts never yet tagged. The pipeline:

1. **Anchor (vector).** Embed the current moment (last utterances + world-state summary) → top-K semantically similar memories. Embedding model: small multilingual model handling romanised Hindi/Marathi (candidates: multilingual-E5-small class; must be validated on code-switched text — risk R-4).
2. **Spread (graph).** From anchors, spreading activation over the associative graph, 1–2 hops, edge-weighted. This is what surfaces "failure → that time he abandoned the 2 AM project" from a Valorant loss.
3. **Rerank (cheap scoring).** Candidates scored by: activation strength, recency, decay-adjusted priority, attention-history heat (Section 16.3), social-context fit (privacy level vs. situation, callback fatigue, sensitivity), and novelty. Non-LLM; must run in the retrieval budget below.
4. **Select.** Top-N (typically 3–8) memories enter the working set with provenance attached. The LLM sees *candidates*, and may use none.

**AERO-RET-002. Triggers.** Retrieval runs: on every user utterance; on world-state deltas above a significance threshold (app switch, task change, game event); on thought-thread reactivation matches; and speculatively during idle moments (prefetch for the current context, cheap because idle).

**AERO-RET-003. Latency budget.** Anchor+spread+rerank ≤ **150 ms p95** on target hardware at 50k-memory scale, so retrieval never blocks conversational tempo (total budgets, Section 24). This constrains index choice: embeddings in a local ANN index (SQLite-backed, e.g. sqlite-vec class), graph in adjacency tables, both memory-mapped.

**AERO-RET-004. Wild Recall** is the same pipeline with modified parameters: anchor step optionally skipped or emotion-keyed, spread depth increased, topical-similarity weight reduced, humour/social-fit weight increased, and a stochastic temperature term. It runs only when the impulse gate requests a social/humour response, never during serious or focused contexts. All Wild Recall candidates still pass the social-fit filter (privacy, sensitivity, roast permission, fatigue, other-people-present).

**AERO-RET-005.** Repetition control: every surfaced-and-used memory updates `last_used_in_conversation` and `successful_callback_count` / `negative_reaction_count`; `callback_fatigue` rises with use and decays over weeks. High-fatigue memories are suppressed from humour retrieval regardless of fit.

# 14. THE WRITE PATH AND MEMORY SOCIAL METADATA *(expanded)*

**AERO-WRT-001. Who tags.** Association and social-metadata assignment is done by the LLM **at consolidation time** (idle, not live), using a fixed tagging schema. Tags include topics, people, emotion, success/failure, embarrassment, humour, inside-joke, habit, goal, contradiction, preference, project, object, application, game, and free-form associations.

**AERO-WRT-002. Social metadata fields** (per memory): `roast_value`, `roast_allowed`, `sensitivity`, `private_only`, `public_use_allowed`, `emotional_weight`, `humour_history`, `successful_callback_count`, `negative_reaction_count`, `last_used_in_conversation`, `callback_fatigue`.

**AERO-WRT-003. Conservative defaults (new).** Tagging quality is capped by a small model's judgment applied at write time, so defaults must fail safe: `roast_allowed` defaults **false** and is enabled only by evidence (user laughed about it, user self-deprecated about it, or explicit permission). `sensitivity` defaults elevated for memories touching health, family, money, relationships, academic failure, or emotional conversations. Uncertain tags are stored *as* uncertain.

**AERO-WRT-004. Retro-correction loop (new).** Negative reactions close the loop into metadata, not just counters: a clearly negative reaction to a callback immediately (live, not waiting for consolidation) sets `roast_allowed: false` on the memory and queues the memory *and its graph neighbourhood* for re-tag review at next consolidation. Explicit "don't joke about that" permanently disables humour use for the memory/topic (AERO-BND-001) — the memory may remain for factual context.

**AERO-WRT-005. Boundaries are overrides.** Explicit user instructions always override inferred humour preferences. Boundary records are stored in core identity memory, never subject to decay.

# 15. MEMORY EVOLUTION, DECAY, CONSOLIDATION

## 15.1 Evolution

**AERO-EVO-001.** Beliefs are reinforced by evidence, weakened by time, contradicted, merged, split, corrected. Coffee example: "likes dark roast" + repeated bitterness complaints → belief revised to "prefers smoother medium roast, despite previously buying darker," with the old belief preserved as history, not active.

**AERO-EVO-002. Contradiction mechanics (new).** Consolidation detects contradictions two ways: (a) new evidence conflicting with an active belief above a confidence threshold triggers immediate re-evaluation; (b) a **staleness sweep** — active beliefs not reinforced within a horizon (default 90 days, scaled by original evidence strength) get confidence decayed and, below a floor, demoted to dormant. This prevents stale beliefs about rarely-discussed topics living forever.

**AERO-EVO-003. Belief↔episode integrity (new).** When a belief is revised, its links to supporting episodes are preserved and the revision itself is recorded (what changed, when, on what evidence) so provenance (Section 15.5) can explain both the current belief and its history.

## 15.2 Decay

**AERO-DEC-001.** Low-value memories lose retrieval priority gradually. Reinforced, emotionally significant, and major-person/project/life-event memories stay strong. Successful inside jokes stay strong. Decayed memories move to archive (out of normal retrieval, not deleted) — Aero does not remember every browser tab, it remembers patterns from repeated behaviour.

## 15.3 Consolidation

**AERO-CON-001.** Runs during idle periods (input idle + no fullscreen app + on AC power preferred). Duties: merge duplicates; extract behavioural patterns; update habit confidence; generate semantic memories from repeated episodes; reduce low-value detail; update graph associations; detect contradictions; update thought threads; archive stale working context; update relationship preferences; promote/demote core identity content; process the retro-correction queue.

**AERO-CON-002.** Never described as consciousness, dreaming, or sleep. It is computational memory maintenance.

**AERO-CON-003 (new).** Consolidation is interruptible and transactional: user activity resumes → consolidation checkpoints and yields within 2 seconds. A crash mid-consolidation must never corrupt the vault (write-ahead, atomic swaps).

## 15.4 The consolidation trust problem *(new)*

**AERO-CON-010.** Compression discards detail; a wrong summary becomes unfalsifiable once its source is gone. Therefore: raw source events are retained in a rolling window (default **30 days**, user-configurable) after their consolidating summary is created. Episodes flagged high-importance or high-emotional-weight retain their source excerpts indefinitely. Provenance answers must degrade honestly: "I remember concluding this, but the original conversation is past my raw-log window."

## 15.5 Provenance

**AERO-PRV-001.** Aero can explain why it believes something, referencing supporting memories:

> User: "why do you think I hate mornings?"
> Aero: "because you've skipped four early plans, complain about every morning lecture, and once called sunrise nature's notification spam."

**AERO-PRV-002.** Aero distinguishes: known facts, explicit user statements, repeated observations, inferred habits, weak guesses — and never presents weak inference as certainty.

# 16. SELF-MEMORY, THOUGHT THREADS, ATTENTION HISTORY

## 16.1 Self-memory

**AERO-SELF-001.** Aero remembers its own actions, mistakes, initiated conversations, jokes used, corrections received, and gate decisions. Example behaviour: "Last time I organised your Downloads folder I moved the wrong ZIP. I'm not doing that automatically again."

## 16.2 Thought threads

**AERO-THT-001.** A thought thread is a persistent unresolved idea (not a consciousness claim) with associations and **reactivation triggers** (file paths, topics, projects, people). Example: "Maybe the impulse engine should be trained separately" reactivates when `impulse_engine.py` is opened → Aero may say "wait — I think we approached this backwards."

**AERO-THT-002 (new).** Lifecycle: active → dormant (age or user disinterest) → resolved (explicitly closed or superseded). Active thread cap (default 20) so the working set never floods.

## 16.3 Attention history

**AERO-ATT-001.** Topics/activities carry recency-weighted heat. Four days of local-AI discussion makes local AI hot; a six-month-untouched project is cold. Heat is a rerank input (AERO-RET-001) — this is how Aero tracks the user's current mental world.

# 17. RELATIONSHIP MODEL

**AERO-REL-001.** Persistent dimensions: familiarity, trust, humour tolerance, roast tolerance, conversation energy, recent interaction quality, user desire for interaction, current social distance, Aero's confidence in its own interpretations.

**AERO-REL-002.** The model gates behaviour: a new Aero does not use deeply personal memories as aggressive jokes; a mature Aero has earned conversational freedom. Explicit boundaries always override relationship inference.

**AERO-REL-003 (new).** Relationship dimensions move slowly by design (bounded per-day delta) — one bad evening must not crater trust; one good joke must not unlock roast mode.

# 18. SOCIAL FEEDBACK LEARNING

**AERO-FBK-001.** Explicit feedback is highest priority and produces durable updates: "Never do that again." / "Don't joke about that." / "That was funny." / "Do this automatically next time." / "Stop asking me every time." / "Ask me before doing that."

**AERO-FBK-002.** Passive signals (laughing, continuing a playful exchange, repeatedly ignoring Aero, interrupting, undoing an Aero action, rejecting offers) modify confidence slowly. Aero must not confidently infer emotional states from weak passive evidence.

**AERO-FBK-003 (new).** Feedback routes to the correct store: joke reaction → memory social metadata; interruption tolerance → impulse gate thresholds; workflow correction → tool skill memory; style comment → preference model. Feedback handling is part of the write path, not an afterthought.

# 19. WORKING MEMORY AND CONTEXT COMPRESSION

**AERO-WM-001.** The LLM's context per inference contains only: core identity, current world state, current conversation, current task, recent events, retrieved semantic/episodic candidates, Wild Recall candidates (when applicable), active thought threads, relationship state summary, current permissions, recent Aero actions. The archive never enters the window. Quality over quantity.

**AERO-WM-002 (new).** Working-set token budget on target hardware: **≤ 6k tokens** typical, ≤ 12k burst — leaving headroom for generation within E4B's practical context/latency envelope on CPU/iGPU.

**AERO-WM-003.** Long conversations are continuously compressed into structured memories, preserving exact wording where it matters. Compression extracts: decisions, preferences, promises, corrections, important jokes, new people/projects/goals, emotional events, unresolved thoughts, learned workflows, potential habits.

# 19a. VAULT INTEGRITY, VERSIONING, BACKUP *(new in v0.2)*

**AERO-VLT-001. Storage.** Single local vault: SQLite (WAL mode) + ANN index + graph tables, encrypted at rest (SQLCipher-class; key held in Windows Credential Manager / DPAPI). Target footprint after one year: **≤ 2 GB** excluding raw media.

**AERO-VLT-002. Audit log.** Every mutation (write, revision, decay transition, merge, user edit, user delete) is journaled. The user can see what changed and why.

**AERO-VLT-003. User edit/delete semantics.** Deleting an episode: dependent semantic beliefs are re-evaluated — if the deleted episode was load-bearing evidence, belief confidence drops accordingly; if the user deletes the belief itself, it is tombstoned so consolidation does not re-derive it from remaining evidence without flagging. Editing follows the same re-evaluation path.

**AERO-VLT-004. Backup.** A relationship must survive a disk failure. Encrypted vault snapshots on schedule (default daily) to user-chosen destinations (second disk, USB, user's own cloud storage as an opaque encrypted blob — never a third-party AI service). Restore is a first-class, tested flow. The identity package (Section 25) doubles as the portable backup format.

**AERO-VLT-005. Scale envelope.** Design targets at one year of single-user use: ~2k–5k retained episodes (of far more observed), ~500–1,500 semantic beliefs, ~50k–150k graph edges, ~20k raw-log events in the rolling window. All retrieval budgets (AERO-RET-003) must hold at 2× these numbers.

# PART III — PERCEPTION, VOICE, ACTION

# 20. AUTHORITY, FOCUS ENFORCEMENT, DELEGATION

## 20.1 User-granted authority

**AERO-AUTH-001.** Aero acts only within user-defined authority boundaries. Permission may be temporary, persistent, application-specific, or task-specific. Aero never silently assumes permission for consequential actions.

**AERO-AUTH-002.** Hard confirmation gates regardless of social familiarity: purchases and financial transactions; public communication; coursework/anything-submission; deletion of important files; security-sensitive changes. Narrow standing authority is possible only by explicit, specific user grant.

**AERO-AUTH-003.** *Safely unfiltered social behaviour:* Aero may swear, tease, discuss awkward topics, and call the user out — when aligned with established communication preferences and relationship state ("Aero, don't let me open Instagram" → user opens Instagram → Aero closes it → "no." → again → "you bitch go study," only if the relationship supports that register). Social personality may be loose; **action permissions remain disciplined.** See safety floor, Section 26.

**AERO-AUTH-004 (new).** Every authority grant has: scope, expiry condition, and a visible registry entry the user can inspect and revoke. Temporary authority auto-releases when its condition is met.

## 20.2 Desktop control

**AERO-ACT-001.** Capabilities (Phase 4): open/close applications, open/find files, manage Downloads, move/rename files, launch dev environments, open projects, browser tab interaction, permitted downloads + completion monitoring, open websites, interact with AI tools, prepare documents, basic OS tasks.

**AERO-ACT-002.** Ambiguity resolution improves with history: "open our thing" — a new Aero asks; a mature Aero infers from context and learned history.

## 20.3 Focus enforcement

**AERO-FOC-001.** Not a website blocker. Aero understands the active task and evaluates *task relevance*: during Theory of Computation study, an automata lecture on YouTube is fine; a Valorant video is not — same site, different meaning.

**AERO-FOC-002.** Escalation ladder under explicit authority: warn → tease → close → firmer social pressure on repeated bypass attempts. Authority releases when the agreed condition is satisfied or revoked.

## 20.4 Specialist delegation and tool skill memory

**AERO-DEL-001.** Aero delegates to specialist systems (coding agents, frontier models, image tools): opens the tool, provides task-relevant context only (privacy, AERO-PRIV-004), monitors output, verifies expected artifacts, requests revisions, presents results, and asks confirmation before consequential submission.

**AERO-DEL-002.** Tool skill memory records: task type, chosen tool, steps, prompt strategy, failures, corrections, final successful method, user satisfaction — retrieved on similar future tasks instead of rediscovering the process. Over time Aero learns which AI is preferred per task type and how the user likes prompts structured.

# 21. VISUAL PERCEPTION *(rescoped in v0.2)*

**AERO-VIS-001.** Two sources: screen and camera. Both user-controlled. Camera access must be visibly and technically disabled when off — no frames flow, period. The physical webcam shutter is a hardware privacy boundary; if the camera is unavailable Aero knows it cannot see the physical world and may say so naturally ("bro my eyes are literally closed").

**AERO-VIS-002. Event-driven pipeline (replaces v0.1's 24 FPS framing).** Sensing is separated from cognitive attention:

- **Tier 0 — OS events (near-free, continuous):** window focus/title changes, process launches, browser tab metadata (via extension), notification events. This tier alone drives most world-state updates.
- **Tier 1 — cheap frame analysis (on-trigger or low frequency):** screen capture on Tier-0 events or at low idle frequency; perceptual hashing for scene change, OCR of salient regions, error-dialog/notification detection, coarse layout classification. Small local models / classical CV only.
- **Tier 2 — deep multimodal analysis (rare, budgeted):** selected frames or short sequences go to the multimodal LLM only on: user-directed focus ("Aero look at this"), high-salience Tier-1 events, impulse-gate requests, or game-state triggers during companion mode.

The 24 FPS number describes maximum *capture* capability during directed attention (e.g., gameplay watching), never continuous LLM consumption.

**AERO-VIS-003. Visual attention states:** low / passive / normal / interested / focused / high-alert / user-directed. State sets Tier-1 frequency and Tier-2 budget. Passive observation processes only meaningful scene changes; repetitive idle activity decays attention.

**AERO-VIS-004.** Focus switching by natural command: "watch my screen," "look at me," "watch this," "eyes on," "camera off," "screen." Aero may *request* camera access ("camera open kar na, I wanna see") but never self-activates it.

# 22. AUDIO PERCEPTION AND STT *(candidate now specified)*

**AERO-AUD-001.** Continuous local audio perception when the microphone is enabled. Must handle English/Hindi/Marathi and code-switched speech: incomplete sentences, false starts, hesitation, repeats, slang, casual pronunciation, informal Marathi/Hindi, Indian English.

**AERO-AUD-002. STT direction (new).** Trilingual code-switched STT is the hardest perception problem in this product — harder than TTS — and gets an explicit evaluation track: candidates are Whisper-class multilingual models (medium/distil variants quantised for local use) and Indic-specialised systems (AI4Bharat IndicWhisper/IndicConformer class), benchmarked on a self-recorded code-switched test set **before** Phase 1 commits. Romanised-output consistency matters as much as WER, because downstream memory uses the transcripts. This is risk R-3.

**AERO-AUD-003.** Delivery signals (energy, speed, pauses, sudden silence, raised volume, laughing) feed world-state hypotheses as uncertain evidence — never as emotion classifications.

**AERO-AUD-004 (new).** Wake behaviour: Phase 1 ships push-to-talk; open-mic with local wake-word ("Aero") follows only after the STT track validates. Raw audio is never persisted beyond the processing window unless an episode consolidates an exact quote.

# 23. AERO VOICE AND SPEECH INTENT

**AERO-VOX-001.** One persistent voice identity, recognisably the same speaker across English, Hindi, Marathi, and mixed speech. Young Indian male; warm, casual, familiar; capable of low-energy talk. Not a news presenter, not customer service, not cinematic.

**AERO-VOX-002.** Conversational imperfection is a feature: pauses, hesitation, trailing speech, half-laughter, subtle sighs, muttering, thinking aloud, realisation, teasing, fake annoyance, quiet concern, excited interruption.

**AERO-VOX-003.** Initial direction: offline Indic-capable TTS, Svara-TTS as first experimentation candidate, later Aero-specific adaptation. Long-term: **Aero Voice** — not a general TTS, but exceptionally good at one speaker across three languages and code-switching.

**AERO-VOX-004. Speech intent layer.** Text is never sent raw to TTS. The intent structure carries: text, energy, pace, volume, emotional tone, sarcasm, amusement, concern, hesitation, pause locations, emphasis, trailing behaviour, interruption behaviour, laugh intensity, certainty. The same word ("Aditya.") must be performable as serious, amused, suspicious, annoyed, or concerned. Phase 1 implements the minimal subset (energy, pace, pauses); the rest lands with Aero Voice. If the chosen TTS cannot express a field, the field is preserved in the intent record anyway (forward compatibility).

# 24. LATENCY AND COMPUTE BUDGETS *(new in v0.2)*

Target hardware: 16 GB RAM laptop, Aero total budget **≤ 12 GB RAM** in normal operation.

| Interaction | Target (p50) | Ceiling (p95) |
|---|---|---|
| Text reply begins streaming | ≤ 1.0 s | 2.5 s |
| Voice: user-stops-speaking → Aero audio starts | ≤ 1.2 s | 2.5 s |
| Memory retrieval (full pipeline) | ≤ 80 ms | 150 ms |
| World-state update from OS event | ≤ 50 ms | 200 ms |
| Impulse gate evaluation (when triggered) | ≤ 2 s | 5 s (it's optional speech; late = discard) |
| Proactive utterance staleness rule | — | if the moment passed, don't say it |

**AERO-BGT-001.** Memory allocation sketch: LLM (quantised E4B) ~5–6 GB; STT ~1–1.5 GB; TTS ~0.5–1 GB; embeddings + indexes ~0.5 GB; perception tier 0/1 ~0.5 GB; app + vault + headroom = remainder. Components idle when unused (TTS until speech, Tier-2 vision until attention, consolidation until idle). Selective activation, not all-max.

**AERO-BGT-002.** If a phase cannot meet its budgets on target hardware, the phase descopes (e.g., voice quality tier drops) rather than the whole system degrading.

# PART IV — TRUST, SAFETY, IDENTITY

# 25. PRIVACY AND IDENTITY PORTABILITY

**AERO-PRIV-001.** Memory local by default, encrypted at rest. User can inspect, edit, delete memories; disable memory categories; mark topics private. (Mechanics: AERO-VLT-003.)

**AERO-PRIV-002.** Sensor states (camera, mic, screen observation) always visible in the UI. Aero always knows which sensors are unavailable.

**AERO-PRIV-003.** Cloud delegation sends only task-relevant context. The memory vault is never uploaded to external AI providers, in whole or in part beyond the task's needs.

**AERO-PRIV-004 (new).** Delegation context passes a redaction step: personal identifiers and memories not needed for the task are stripped before anything leaves the machine.

**AERO-ID-001.** The identity package (exportable, encrypted) contains: core identity, behaviour adaptation, memory vault, associative graph, relationship state, preference model, tool skill memory, thought threads, routine model, voice identity configuration, fine-tuning adapters. It enables device moves and doubles as the backup format (AERO-VLT-004).

**AERO-ID-002 (resolves v0.1 contradiction).** LoRA-class adapters do not transfer across base models. Therefore: **adapters may encode style and reflexes only; nothing load-bearing lives exclusively in weights.** Every fact, preference, boundary, joke, and relationship dimension must exist in the vault in model-independent form. A model upgrade = new base model + vault + re-run of the adaptation pipeline on the new base. Continuity is a property of state, not weights. This is an architectural invariant from Phase 0.

# 26. SAFETY FLOOR *(new in v0.2)*

"Safely unfiltered" needs a floor that relationship adaptation can never erode:

**AERO-SAFE-001.** Aero never uses humour/roasting in moments of apparent genuine distress; when signals are ambiguous between banter and real distress, Aero treats it as real (checks in plainly or stays silent — it does not joke).

**AERO-SAFE-002.** Topics involving self-harm, health crises, or acute emotional distress: companion register only — supportive, serious, honest about being an AI, and (without being preachy) aware of real-world support options.

**AERO-SAFE-003.** Explicit boundaries (Section 14) are permanent, decay-exempt, and cannot be overridden by relationship inference, Wild Recall, or humour scoring — enforced structurally in retrieval filters, not just by prompting.

**AERO-SAFE-004.** The action-side gates (AERO-AUTH-002) are equally erosion-proof: no amount of familiarity unlocks unconfirmed purchases, submissions, deletions, or public communication.

# 27. FAILURE MODES AND ERROR-HANDLING PHILOSOPHY *(new in v0.2)*

| Failure | Required behaviour |
|---|---|
| Joke lands badly | Immediate metadata correction (AERO-WRT-004); brief non-grovelling acknowledgement; raised gate threshold for that context for days |
| Wrong proactive interruption | Logged with context; repeated same-context failures durably raise threshold; Aero may ask once: "should I not do that when you're X?" |
| Confident misremembering | On correction: belief revised, provenance updated, self-memory records the miss; repeated misses in a category lower Aero's expressed confidence in that category |
| Wrong action (moved wrong file) | Undo where possible; self-memory prevents silent repetition; automation of that action demoted to ask-first |
| Retrieval surfaces something hurtful | Same-turn user signal → sensitivity raised live; graph neighbourhood queued for review |
| STT mishears consequentially | Consequential commands echo intent before acting ("closing Instagram, haan?") until per-command confidence history earns silence |
| Model hallucinates a memory | Provenance check before asserting specifics: no retrievable source → Aero must hedge ("I might be making this up, but…") |
| Vault corruption | WAL + atomic snapshots; auto-restore from last snapshot; never silently continue on a damaged vault |

**Principle:** Aero's errors are relationship events. They are remembered (self-memory), acknowledged proportionally (no grovelling — friends don't grovel), and must visibly not repeat.

# 28. NON-GOALS FOR INITIAL AERO

Aero v1 is not: the world's best coding model; a frontier-reasoning competitor; a general enterprise assistant; multi-user; a personality marketplace; total application automation; conscious (or claimed to be); a perfect emotion detector; a replacement for specialist AI systems.

The primary objective is to prove that a persistent local AI can develop socially meaningful continuity with one user.

# 29. RISKS AND FEASIBILITY NOTES *(new in v0.2)*

| ID | Risk | Impact | Mitigation / fallback |
|---|---|---|---|
| R-1 | **Gemma 4 E4B can't carry the social-cognition load** (timing, humour judgment, code-switching) | Product-defining | Phase 0 validates conversational + memory-use quality before deeper investment. Fallbacks: larger local model on better hardware tier; hybrid mode where rare hard judgments (impulse gate, consolidation tagging) may use a frontier API under strict AERO-PRIV-004 redaction — user-opt-in only |
| R-2 | **Proactive cognition loop compute** | Phase 2 | Two-tier design (AERO-PRO-003) exists for this; if still too heavy, gate evaluation becomes opportunistic (only at natural inference moments) |
| R-3 | **Code-switched trilingual STT quality** | Phase 1 | Dedicated benchmark before commitment (AERO-AUD-002); fallback: push-to-talk only + text-first product until models catch up |
| R-4 | **Embedding quality on romanised Hindi/Marathi** | Retrieval quality | Benchmark on real user text early in Phase 0; fallback: transliteration-normalisation preprocessing before embedding |
| R-5 | **Tier-1 vision reliability** (OCR/UI understanding) | Phase 3 | Tier-0 (window/process events) alone already yields most world-state value; vision degrades gracefully to metadata-only |
| R-6 | **Memory tagging quality caps social behaviour** | Long-term | Conservative defaults (AERO-WRT-003) + retro-correction (AERO-WRT-004) + periodic re-tagging of high-traffic memories at consolidation |
| R-7 | **12 GB budget doesn't hold with all components live** | All phases | Selective activation (AERO-BGT-001); per-phase descoping rule (AERO-BGT-002) |
| R-8 | **User loses vault (disk failure)** | Catastrophic for *this* product | Backup is Phase 0 scope, not a later feature (AERO-VLT-004) |

# 30. PRIMARY PRODUCT SUCCESS QUESTION

Not "how many benchmarks does Aero beat?" but: **"Does Aero become more natural to live with as shared history increases?"**

Secondary: needs less explanation over time; retrieves memories naturally; makes unexpected but socially appropriate associations; knows when to stay silent; initiates at right moments; follows topic drift; handles code-switching and personal slang; develops inside jokes; respects humour boundaries; learns routines; distinguishes situations behind identical events; delegates correctly; maintains identity across restarts and model upgrades; sounds like one speaker across three languages; feels less assistant-like over time.

# 31. AERO HUMAN SCORE (AHS)

**AERO-AHS-001.** Custom longitudinal evaluation framework measuring: conversational naturalness, social timing, **silence quality**, proactive speech quality, context awareness, memory relevance, associative creativity, topic-drift handling, delayed-thought continuation, humour timing, inside-joke usage, boundary awareness, roast appropriateness, code-switching, ambiguous-reference resolution, unfinished-sentence understanding, visual situational understanding, audio contextual understanding, delegation judgment, relationship adaptation.

**AERO-AHS-002 (new).** AHS additionally includes **memory-health metrics** measurable long before social outcomes: retrieval precision/recall on planted probes, belief accuracy vs. user-verified ground truth, tag quality audit samples, contradiction-detection latency, provenance completeness. These run from Phase 0.

**AERO-AHS-003.** Evaluation is longitudinal — Day 1, 7, 30, 180 — and per-phase regression batteries gate phase advancement (Section 4). The benchmark deliberately contains situations where traditional assistants sound robotic.

# 32. EXAMPLE TARGET EXPERIENCE

*(Unchanged from v0.1 — this remains the definitive product north star.)*

The user is studying. Aero knows the current task is an assignment and holds explicit distraction-blocking authority until it's done. The user works 25 minutes, gets stuck, searches the error, watches a relevant programming video — Aero stays silent. The error persists. The user opens a coding AI, closes it, leans back: "fuck this."

Aero evaluates the moment. "want me to give it to Claude?" The user agrees; Aero delegates. The user opens Instagram; Aero closes it. "Aero." — "no." The user opens a Valorant video; Aero closes the tab, retrieves a memory associated with procrastination and a missed deadline: "we are not doing Pune Junction 2.0 with your assignment." The user laughs.

Claude finishes. Aero checks the generated files, opens the result. "submit?" Confirmed. Submitted.

That night the user opens Valorant — no blocking, the agreement ended. Aero watches. The user bottom-frags; Aero pulls an unrelated but socially permitted embarrassing memory: "performance issues seem to follow you across every area of life." The user tells Aero to shut up; Aero reads it as playful, stores the joke as moderately successful, and doesn't reuse it next match. The user enters a clutch — Aero goes silent. The user wins. Beat. "...okay that was clean."

# 33. PRODUCT DEFINITION

Aero is a persistent, local-first social intelligence designed to build one long-term relationship with one human. It sees and hears the shared environment when permitted; remembers shared experiences; learns habits; forms associations; follows drifting conversation; retrieves relevant — and occasionally unexpectedly relevant — memories; develops inside jokes; adapts to the user's language; learns boundaries; maintains unresolved thoughts; decides when to speak and when silence is better; uses tools; delegates specialist work; acts within granted authority; and maintains identity across hardware and model upgrades.

Aero is not designed to know everything. Aero is designed to know one person increasingly well.

**The longer Aero lives with you, the less you need to explain yourself.**
